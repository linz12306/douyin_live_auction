## 一、用户系统

- **角色**：商家账号 / 用户账号独立，注册时选择角色，不可切换
- **登录**：用户名 + 密码，JWT token 认证
- **资料**：昵称 + 头像（上传图片），排行榜展示用
- **虚拟余额**：注册赠送 100 万虚拟币，出价时**预扣冻结**（非直接扣款）

### 余额冻结机制（核心！）

```
出价 → 冻结出价金额（balance -= X, frozen_amount += X）
被超越 → 自动解冻（frozen_amount -= X, balance += X）
成交 → 转为实际扣款（frozen_amount -= X）
取消 → 全部解冻
```

- 余额不足时：出价按钮置灰 + Toast 提示"余额不足"
- 不可提现，仅为竞拍使用
- 允许同时参与多个竞拍，余额按商品独立冻结

## 三、竞拍规则

| 规则 | 说明 |
|------|------|
| 起拍价 | 默认 0 元（0元起拍） |
| 加价幅度 | **双模式**：固定金额（如每次+10元）或 百分比（如每次+5%），商家选择 |
| 封顶价 | 商家可选设置，达到后自动成交；NULL 表示不封顶 |
| 竞拍时长 | 固定时长（如60s / 5min / 30min），商家设定 |
| 自动延时 | **Soft Close 机制**：每次有人出价，重置倒计时到 N 秒，最多延长 M 次 |

### 自动延时规则（Soft Close）

```
设定：auto_extend_seconds = 15s, max_extend_count = 5
流程：
  - 竞拍剩余 ≤ 15s 时有人出价 → 倒计时重置为 15s
  - 每次延时 current_extend_count += 1
  - 达到 max_extend_count 后不再延时，最后一人出价即成交
```

### 加价幅度约束

- 固定模式：金额 ≥ 1 元
- 百分比模式：1% ~ 20%
- 后端校验出价金额是否满足：`amount ≥ current_price + calculate_increment(current_price)`

## 四、竞拍状态机（5 状态）

```
┌─────────┐     达到开始时间      ┌────────┐
│ pending  │ ──────────────────→ │ active │
│ (待开拍) │                      │(进行中)│
└────┬─────┘                      └───┬────┘
     │                                │
     │ 商家取消                        │ 有人出价 → 时间到/封顶价达到
     ↓                                ↓
┌──────────┐                   ┌──────────────┐
│cancelled │                   │ ended_sold   │
│ (已取消)  │                   │ (已成交)      │
└──────────┘                   └──────────────┘
                                    ↑
     ┌──────────────────────────────┘
     │ active → 无人出价 → 时间到
     ↓
┌───────────────┐
│ ended_no_bid  │
│ (流拍)         │
└───────────────┘
```

### 状态说明

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `pending` | 待开拍 | 商品上架成功 |
| `active` | 进行中 | 到达开始时间 |
| `ended_sold` | 已成交 | 有人出价 + 时间到或封顶价达到 |
| `ended_no_bid` | 流拍 | 无人出价 + 时间到 |
| `cancelled` | 已取消 | 商家取消 |

## 五、出价 & 成交流程

### 出价两阶段流程（核心优化！）

```
阶段一：锁内（3-5ms，串行，保证数据一致性）
  1. Redis SETNX 抢锁（auction:{id}:bid_lock, TTL=5s）
  2. 内存校验：竞拍状态、出价金额
  3. DB 乐观锁：冻结出价者余额
  4. 插入 bid 记录 + 更新旧出价状态
  5. 更新 auctions 表 + 内存状态
  6. 标记异步事件 + 释放 Redis 锁

阶段二：异步（goroutine，毫秒级延迟可接受）
  7. 检查封顶价 → 自动成交
  8. 检查 Soft Close → 延时
  9. WebSocket 广播价格 + 私信被超越者
```

### 成交流程

```
时间到/封顶价达到
  → 竞拍状态 = ended_sold
  → 生成订单（status=pending_confirm）
  → 通知中标者
  → 买家 30 分钟内确认
     → 确认：status=pending_payment → 模拟支付 → status=paid
     → 不确认：status=cancelled（confirm_timeout），余额全部解冻
```

### 取消流程

- **pending 阶段**：商家可直接取消
- **active 阶段**：
  - 最后出价后 30 秒内**不可取消**（防止价格低商家跑路）
  - 需二次确认 + 填写取消原因
  - 取消后所有出价者的冻结余额自动解冻
  - 所有出价记录 status 改为 cancelled
- **已结束**：不可取消

## 六、前端数据流规则（重要设计决策）

### WS 是唯一真理源

```
WebSocket（实时流） ────▶ Zustand Store ────▶ React 组件
REST API（初始化） ────▶ 只加载一次，之后被 WS 覆盖
```

- 页面首次打开：REST API 加载初始数据到 store
- WebSocket 连接建立后：所有实时数据以 WS 为准
- WS 数据自动覆盖 REST 数据（防止 UI 闪烁）
- 用户自己发起的出价：REST POST 出价 → 等待 WS 广播回包 → 不信任 REST 返回值

### 时钟同步规则

- 服务端时间戳（server_time）随每条 WS 消息下发
- 客户端计算时间差 offset = server_time - client_send_time
- 倒计时 = end_time - Date.now() + offset
- 倒计时用 requestAnimationFrame 刷新（60fps）

## 九、可观测性要求（新增）

### 审计日志
- 每次竞拍状态变更记录到 `auction_logs` 表
- 记录内容：操作的 auction_id、操作类型（出价/冻结/解冻/成交/取消/延时）、操作详情（JSON）、操作者
- 作用：评委审查、bug 排查、数据对账

### 健康检查
- 提供 `/healthz` 接口，返回后端各组件状态（DB / Redis / 竞拍引擎）
- 影响：评委可在演示时检查系统健康度

### 结构化日志
- 使用 zerolog（Go 的零分配结构化日志库）
- 关键操作记录：出价请求、状态变更、WS 连/断、定时器执行、异常情况

### 竞拍引擎指标
- 活跃竞拍数
- 出价请求量/成功率/平均耗时
- WebSocket 连接数
- 锁竞争率

## 十一、并发 & 高可用目标

### 并发目标
- 支持 1000 人同时在线
- WebSocket 心跳保活（30s）+ 自动重连 + 断线补发状态快照

### 分布式锁
- Redis SETNX 保证出价串行化
- DB 乐观锁作为降级方案
- 锁内仅做最小操作（3-5ms），广播异步化

### 内存状态
- 活跃竞拍加载到内存（sync.Map + AuctionState）
- 出价校验走内存（1μs），不每次读 MySQL
- 服务重启时从 DB 恢复

### 异常降级策略

| 场景 | 方案 |
|------|------|
| Redis 挂了 | 降级到 DB 乐观锁 |
| WebSocket 断线重连 | 补发状态快照（价格+排名+剩余时间） |
| 后端重启 | 从 DB 恢复竞拍状态到内存 |
| 倒计时同步 | 服务端 server_time + 客户端差值修正 |
| 数据库写入失败 | 余额操作失败则出价拒绝，返回友好提示 |
