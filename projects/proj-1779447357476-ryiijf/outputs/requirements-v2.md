# 需求确认文档 v2.0（魔鬼产品经理修订版）

> 基于 v1.0 经过产品审查后的完整修订版，补充了冻结余额机制、状态机细化、边界条件等。

---

## 一、用户系统

- **角色**：商家账号 / 用户账号独立，注册时选择角色，不可切换
- **登录**：用户名 + 密码，JWT token 认证
- **资料**：昵称 + 头像（上传图片），排行榜展示用
- **虚拟余额**：注册赠送 100 万虚拟币，出价时**预扣冻结**（非直接扣款）

### 余额冻结机制（核心！）

```
出价 → 冻结出价金额（balance -= X, frozen_amount += X）
被超越 → 自动解冻（frozen_amount -= X, balance += X）
成交 → 转为实际扣款（frozen_amount -= X）
取消 → 全部解冻
```

- 余额不足时：出价按钮置灰 + Toast 提示"余额不足"
- 不可提现，仅为竞拍使用
- 允许同时参与多个竞拍，余额按商品独立冻结

## 三、竞拍规则

| 规则 | 说明 |
|------|------|
| 起拍价 | 默认 0 元（0元起拍） |
| 加价幅度 | **双模式**：固定金额（如每次+10元）或 百分比（如每次+5%），商家选择 |
| 封顶价 | 商家可选设置，达到后自动成交；NULL 表示不封顶 |
| 竞拍时长 | 固定时长（如60s / 5min / 30min），商家设定 |
| 自动延时 | **Soft Close 机制**：每次有人出价，重置倒计时到 N 秒，最多延长 M 次 |

### 自动延时规则（Soft Close）

```
设定：auto_extend_seconds = 15s, max_extend_count = 5
流程：
  - 竞拍剩余 ≤ 15s 时有人出价 → 倒计时重置为 15s
  - 每次延时 current_extend_count += 1
  - 达到 max_extend_count 后不再延时，最后一人出价即成交
```

### 加价幅度约束

- 固定模式：金额 ≥ 1 元
- 百分比模式：1% ~ 20%
- 后端校验出价金额是否满足：`amount ≥ current_price + calculate_increment(current_price)`

## 四、竞拍状态机（5 状态）

```
┌─────────┐     达到开始时间      ┌────────┐
│ pending  │ ──────────────────→ │ active │
│ (待开拍) │                      │(进行中)│
└────┬─────┘                      └───┬────┘
     │                                │
     │ 商家取消                        │ 有人出价 → 时间到/封顶价达到
     ↓                                ↓
┌──────────┐                   ┌──────────────┐
│cancelled │                   │ ended_sold   │
│ (已取消)  │                   │ (已成交)      │
└──────────┘                   └──────────────┘
                                    ↑
     ┌──────────────────────────────┘
     │ active → 无人出价 → 时间到
     ↓
┌───────────────┐
│ ended_no_bid  │
│ (流拍)         │
└───────────────┘
```

### 状态说明

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `pending` | 待开拍 | 商品上架成功 |
| `active` | 进行中 | 到达开始时间 |
| `ended_sold` | 已成交 | 有人出价 + 时间到或封顶价达到 |
| `ended_no_bid` | 流拍 | 无人出价 + 时间到 |
| `cancelled` | 已取消 | 商家取消 |

## 五、出价 & 成交流程

### 出价流程

```
1. Redis 抢锁（SETNX auction:{id}:bid_lock）
2. 校验：金额 ≥ 当前价 + 加价幅度
3. 校验：用户余额 ≥ 出价金额
4. 冻结出价金额（DB 乐观锁更新 wallet）
5. 插入 bid 记录（status='active'）
6. 旧最高出价者的 bid 状态改为 'outbid'，余额解冻
7. 更新 auctions 表：current_price, highest_bidder_id
8. WebSocket 广播新价格 + 排名变化
9. 释放 Redis 锁
```

### 成交流程

```
时间到/封顶价达到
  → 竞拍状态 = ended_sold
  → 生成订单（status=pending_confirm）
  → 通知中标者
  → 买家 30 分钟内确认
     → 确认：status=pending_payment → 模拟支付 → status=paid
     → 不确认：status=cancelled（confirm_timeout），余额全部解冻
```

### 取消流程

- **pending 阶段**：商家可直接取消
- **active 阶段**：
  - 最后出价后 30 秒内**不可取消**（防止价格低商家跑路）
  - 需二次确认 + 填写取消原因
  - 取消后所有出价者的冻结余额自动解冻
  - 所有出价记录 status 改为 cancelled
- **已结束**：不可取消

## 九、并发 & 高可用目标

### 并发目标
- 支持 1000 人同时在线
- WebSocket 心跳保活（30s）+ 自动重连 + 断线补发状态快照

### 分布式锁
- Redis SETNX 保证出价幂等性（同一用户不能同时出两笔）
- DB 乐观锁作为降级方案

### 异常降级策略

| 场景 | 方案 |
|------|------|
| Redis 挂了 | 降级到 DB 乐观锁 |
| WebSocket 断线重连 | 补发状态快照（价格+排名+剩余时间） |
| 后端重启 | 从 DB 恢复竞拍状态，Redis 重新初始化 |
| 倒计时同步 | 服务端为准，客户端仅展示服务端下发的时间 |
| 数据库写入失败 | 余额操作失败则出价拒绝，返回友好提示 |

## 十、关键业务规则汇总

| # | 规则 | 细节 |
|---|------|------|
| 1 | 0 元起拍 | 默认起拍价 0 |
| 2 | 加价幅度 | 双模式（固定/百分比），后端校验 |
| 3 | 封顶价 | 可选，达到自动成交 |
| 4 | Soft Close 延时 | 出价重置倒计时，有最大次数 |
| 5 | 余额预扣冻结 | 出价冻结→被超越解冻→成交扣款 |
| 6 | 多商品并发出价 | 同时参与多个竞拍，余额独立冻结 |
| 7 | 取消三级限制 | pending 直接取消 / active 需确认 / 最后出价30s内不可取消 |
| 8 | 成交确认超时 | 30分钟不确认自动流拍 |
| 9 | 分布式锁 | Redis SETNX + DB 乐观锁双保险 |
| 10 | 断线重连 | 补发完整状态快照 |
