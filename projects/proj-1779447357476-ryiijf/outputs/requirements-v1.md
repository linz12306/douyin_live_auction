## 一、用户系统

### 角色与认证
- **分离角色**：商家账号 / 用户账号独立注册、独立登录
- 登录方式：用户名 + 密码，JWT token 认证
- 资料：昵称 + 头像（上传图片），排行榜展示用
- 一个人可以同时是商家和用户（不同账号）

### 虚拟余额系统
- 注册赠送 **100 万虚拟币**（好听+大气）
- **预扣（冻结）模式**（标准拍卖做法）：
  - 出价时：冻结出价金额
  - 被超越时：自动解冻，余额恢复
  - 成交时：冻结金额转为实际扣款
- 余额不足 → 出价按钮置灰，显示"余额不足"
- 不能提现，仅限站内竞拍使用
- MVP 阶段不做充值入口，如时间允许可后续补充

## 三、竞拍规则

### 基础规则
- 起拍价：默认 0 元（0 元起拍）
- 竞拍时长：商家设定固定秒数（如 60s / 5min / 30min）
- 封顶价：**商家可选**设置，达到后自动成交给最后出价者

### 加价幅度（双模式）
- **模式 A：固定幅度** —— 商家设一个固定值（如 10 元），每次加价 ≥ 当前价 + 该值
- **模式 B：百分比幅度** —— 每次加价 ≥ 当前价 × (1 + 百分比)，适合高价商品
- 商家在发布时选择模式并配置参数

### 自动延时（Soft Close 软截止机制）
- **规则**：结束前 N 秒内有人出价，倒计时重置为 N 秒
- **可配置参数**：
  - 延时时长：10-30 秒（商家设定）
  - 最大延长次数：商家设定（如最多延长 5 次）
- **超出最大次数后**：最后一次出价者直接成交
- 示例：设延时 15 秒、最多延长 3 次 → 总延时上限 45 秒，之后不再延长

### 校验规则（边界条件）
- 加价幅度合理性：固定模式下幅度 > 0，百分比模式下 0% < 比例 ≤ 50%
- 竞拍时长：最短 30 秒，最长 1 小时
- 封顶价 > 起拍价（如果设了封顶价）

## 四、竞拍状态机（完整）

```
                    ┌──────────┐
                    │  pending │  （待开拍）
                    └────┬─────┘
                         │
                    商家开始竞拍
                         │
                         ▼
                    ┌──────────┐
                    │  active  │  （进行中）
                    └──┬───┬──┘
                       │   │
              时间到/封顶  商家取消
                       │   │
              ┌────────┘   └────────┐
              ▼                      ▼
    ┌─────────────┐         ┌───────────┐
    │ ended_sold  │         │ cancelled │
    │ （已成交）   │         │ （已取消） │
    └─────────────┘         └───────────┘
    ┌─────────────┐
    │ended_no_bid │
    │ （流拍）     │
    └─────────────┘
```

### 5 个状态定义
| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `pending` | 待开拍 | 商品发布成功 |
| `active` | 进行中 | 商家手动开始 / 定时自动开始 |
| `ended_sold` | 已成交 | 时间到且有人出价 / 达到封顶价 |
| `ended_no_bid` | 流拍 | 时间到且无人出价 |
| `cancelled` | 已取消 | 商家取消（受限制，见下文） |

### 商家取消限制
- **仅 pending 状态**：可直接取消，无需审批
- **active 状态（有人出价）**：需二次确认 + 填写取消原因 + 公示取消原因
- **active 状态（最后出价后 30 秒内）**：不可取消（防止看价太低耍赖）
- 取消后：所有冻结余额立即解冻，WebSocket 广播取消通知

## 五、出价 & 成交流程

### 出价逻辑
1. 用户点击出价
2. 后端校验：
   - 竞拍状态 = active
   - 出价金额 ≥ 当前价 + 加价幅度
   - 用户余额 ≥ 出价金额（冻结额度）
3. **Redis 分布式锁**获取 → 乐观锁更新 → 释放锁
4. 冻结出价金额
5. 解冻前一个最高出价者的金额
6. 广播新价格 + 排名变化

### 成交流程
1. 时间到 / 封顶价达到 → 状态变为 `ended_sold`
2. WebSocket 广播"竞拍结束，恭喜 xxx 以 xxx 元成交"
3. 中标者收到确认通知
4. **中标者有 30 分钟确认时间**
5. 30 分钟内确认 → 生成订单 → 进入模拟支付
6. **30 分钟超时未确认 → 流拍** → 退回冻结余额 → 商品可重新上架

### 流拍处理
- 自动流拍后，商品状态变为 `ended_no_bid`
- 商家可选择重新上架（新竞拍）

## 六、商家端（PC 管理后台）

### 竞拍发布
- 商品名称、**多图上传**（最多 9 张）、商品介绍（富文本或 Markdown）
- 竞拍规则配置：
  - 起拍价
  - 加价模式（固定/百分比）+ 幅度值
  - 封顶价（可选，留空=不封顶）
  - 竞拍时长（秒数）
  - 自动延时时长 + 最大延长次数

### 商品管理
- 列表展示四种状态筛选：待开拍 / 进行中 / 已成交 / 流拍 / 已取消
- 支持修改**未开始**竞拍的规则
- 支持取消竞拍（受上述限制）
- 查看每个商品的出价记录、参与人数

### 订单管理
- 成交订单列表：买家、成交价、成交时间、订单状态
- 模拟发货状态更新

### 数据看板（仅本商家数据）
- 成交趋势图（按天/周/月）
- 出价分布直方图
- 用户活跃度（参与人数、人均出价次数）
- 商品维度的成交率统计

## 八、技术架构

### 技术栈
- 后端：Go 1.24 + Gin + gorilla/websocket
- 前端：React 18 + TypeScript + Vite + TailwindCSS
- 数据库：MySQL 8.0（Docker，端口 3306）
- 缓存/锁：Redis 7（Docker，端口 6379）
- 实时通信：WebSocket 长连接，房间级隔离
- 开发规范：SuperSpec（OpenSpec + Superpowers）

### 架构原则
- 分层架构：Handler → Service → Repository
- RESTful API（业务操作） + WebSocket（实时推送）双通道
- Redis：缓存热点数据 + 分布式锁 + 余额冻结/解冻
- MySQL：核心业务数据持久化

## 九、并发目标 & 高可用

### 性能目标
- 支持 **1000 人同时在线**（单房间）
- 出价响应延迟 < 200ms
- WebSocket 推送延迟 < 100ms

### 高可用措施
- WebSocket 心跳保活（30 秒间隔）+ 断线自动重连（指数退避）
- 断线重连后**补发错过的状态快照**（当前价格、排名、剩余时间）
- Redis 分布式锁保证出价幂等性
- 后端重启后从 Redis + MySQL 恢复竞拍状态
- 服务器时钟同步：倒计时以服务端为准，客户端仅做显示

### 异常处理策略
| 场景 | 降级方案 |
|------|---------|
| MySQL 不可用 | Redis 缓存兜底读，写操作排队等恢复 |
| Redis 不可用 | 降级为数据库直查，锁机制降级为乐观锁 |
| WebSocket 断线 | 客户端指数退避重连，重连后拉取状态快照 |
| 后端重启 | 从 Redis + DB 恢复内存中的竞拍状态 |
| 并发出价冲突 | Redis 分布式锁 + DB 乐观锁双保险 |
