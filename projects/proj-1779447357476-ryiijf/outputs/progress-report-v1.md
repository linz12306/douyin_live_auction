## 📋 项目概览


### 核心模块划分
1. **用户认证 + 钱包**（user-auth）
2. **商品管理 + 规则配置**（product-crud）
3. **竞拍引擎 + 状态机**（auction-engine）⭐ 核心
4. **WebSocket 实时通信**（ws-realtime）⭐ 亮点
5. **商家后台前端**（merchant-admin）
6. **订单 + 支付 + 看板**（order-system）

### 关键技术决策
| 决策点 | 选择 | 原因 |
|--------|------|------|
| 后端语言 | Go | 高并发场景性能好，goroutine适合WebSocket |
| 余额机制 | 预扣冻结 | 拍卖行业标准，避免超额出价 |
| 并发控制 | Redis锁 + DB乐观锁 | 双重保险，Redis抢锁失败时降级到DB |
| 状态机 | 5状态设计 | 区分成交/流拍/取消，业务更清晰 |
| 加价模式 | 双模式（固定+百分比） | 适应不同商品价值场景 |
| 延时机制 | Soft Close | 经典拍卖延时规则，防止最后一秒狙击 |

## 🔗 核心链路

### 主流程（Happy Path）
```
用户注册 → 登录获取JWT → 浏览商品列表 → 进入竞拍详情
→ 出价（冻结余额）→ 实时排名更新 → 竞拍结束
→ 中标确认 → 模拟支付 → 订单完成
```

### 竞拍状态机（5状态）
```
pending（待开拍）
    ↓ 到达开始时间
active（进行中）
    ↓ 有人出价且达到封顶价 → ended_sold（已成交）
    ↓ 时间到且有人出价 → ended_sold（已成交）
    ↓ 时间到且无人出价 → ended_no_bid（流拍）
    ↓ 商家取消（有限制条件）→ cancelled（已取消）
pending → cancelled（未开拍可直接取消）
```

### 出价冻结流程（关键）
```
1. Redis抢锁（SETNX auction:{id}:bid_lock）
2. 校验：金额 ≥ 当前价 + 加价幅度
3. 冻结：用户钱包 balance -= 金额, frozen += 金额
4. 记录：wallet_records 插入冻结记录
5. 出价：bids 表插入新记录 status='active'
6. 解冻：旧出价者钱包 frozen -= 旧金额, balance += 旧金额
7. 更新：auctions 表 current_price, highest_bidder_id
8. 广播：WebSocket 通知所有客户端
```

## 🏗️ 技术架构详解

### 后端分层架构
```
Router (路由层)
  → Middleware (认证、日志、CORS)
    → Handler (HTTP处理、参数校验)
      → Service (业务逻辑、状态机)
        → Repository (数据访问)
          → MySQL (持久化) / Redis (缓存/锁)
```

### WebSocket 设计
- **Hub 模式**：中心化连接管理
- **Room 隔离**：每个竞拍一个房间，支持多直播间扩展
- **心跳保活**：30秒 ping/pong
- **断线重连**：重连后补发状态快照（价格+排名+剩余时间）
- **消息类型**：price_update / outbid / extended / auction_end / bid_success

### 并发安全设计
```
Redis 分布式锁（第一道防线）
  ↓ 抢锁失败
MySQL 乐观锁（version字段，第二道防线）
  ↓ 更新失败
返回错误，用户重试
```

### 高并发优化点
- Redis 缓存热门商品数据（防击穿）
- 数据库连接池调优
- WebSocket 消息压缩（gzip）
- 出价幂等性保证（同一用户500ms内重复出价被拒绝）

## 🧠 个人思考与决策记录

### 为什么选择Go而不是Node.js？
1. **高并发场景**：Go的goroutine比Node.js的event loop更适合处理大量WebSocket连接
2. **性能**：Go编译型语言，运行时性能更好
3. **学习价值**：比赛评分有25%技术深度，Go更能体现技术选型能力
4. **生态**：Gin框架成熟，gorilla/websocket是WebSocket标准库

### 为什么用SuperSpec而不是其他规范工具？
1. **整合度高**：OpenSpec（规划）+ Superpowers（执行）开箱即用
2. **适合1人开发**：不需要复杂的团队协作流程
3. **与牛马AI集成好**：已配置好命令和技能

### 关键技术决策思考
1. **余额冻结 vs 直接扣款**：冻结更安全，符合拍卖行业标准，避免"出价后退款"的复杂逻辑
2. **5状态机 vs 3状态机**：5状态能区分"成交"和"流拍"，订单系统需要这个信息
3. **Soft Close延时**：防止最后一秒狙击，让竞拍更公平
4. **双模式加价**：固定模式简单，百分比模式适合高价值商品

### 时间管理策略
- **前5天全力后端**：先跑通核心闭环，前端可以mock
- **第6-10天双端并行**：前后端同时推进，用API文档对接
- **最后4天打磨**：优先级从高到低，P3功能可砍
