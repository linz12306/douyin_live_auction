## Implemented

- Backend order module:
  - `GET /api/v1/orders`
  - `GET /api/v1/orders/:id`
  - `POST /api/v1/orders/:id/confirm`
  - `POST /api/v1/orders/:id/pay`
  - `POST /api/v1/orders/:id/cancel`
- Order timeout worker runs about once per minute and cancels aged `pending_confirm` orders.
- Buyer flow:
  - `pending_confirm -> pending_payment`
  - `pending_payment -> paid`
  - `pending_confirm -> cancelled` with refund on buyer cancel or confirm timeout.
- Merchant flow:
  - merchant-scoped order list/detail inspection.
- Frontend:
  - user routes `/app/orders` and `/app/orders/:id`.
  - merchant routes `/merchant/orders` and `/merchant/orders/:id`.
  - entries from auction lobby, live room sold state, and merchant product list.
- E2E:
  - `tests/e2e/order-system.spec.ts` covers buyer win -> order list/detail -> confirm -> simulated payment -> paid.

## Key Decision

- Existing auction settlement already deducts the winning frozen amount when creating `pending_confirm` orders.
- Therefore simulated payment is status-only and does not mutate wallet fields.
- Buyer cancellation or confirmation timeout refunds by adding `orders.amount` back to `users.balance`, guarded by locked `pending_confirm` state so refund happens once.
- Order cancellation does not rewrite `auctions` or `products`; auction history remains `ended_sold`.
